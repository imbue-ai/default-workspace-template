- Messaging an agent (`POST /api/agents/{id}/message`) no longer runs a fresh mngr discovery on every send. `AgentManager` now retains each agent's location (host/provider) from the `mngr observe` discovery stream it already consumes -- rebuilt on every full-snapshot event, updated incrementally when an agent-discovered delta arrives, and dropped on destroy events -- and exposes `get_agent_matches_by_id`. `agent_discovery.send_message` consults that live, event-fed state first and messages the agent directly via mngr's pre-resolved `agents_to_message` API; it falls back to a full discovery only when the location is unknown or when the known location no longer reaches the agent. The result is fresh on both agent creation and removal, and discovery stays as a correctness backstop.
- A just-created agent is located the instant its agent-discovered delta arrives, not only at the next full snapshot, so the *first* message to a freshly-created agent also skips discovery. The delta itself omits the host's name, so the location is recorded only when the host is already known from another agent on it (the usual case); a first-agent-on-a-never-seen host stays discovery-resolved until the next snapshot.
- Messaging now addresses agents **by id only**. `send_message(agent_id: AgentId, ...)` no longer accepts a name, so a message always targets the exact, globally-unique agent rather than fanning out across same-named agents on other workspaces/hosts (which is never an intended behavior). The endpoint passes the id it already has (O(1) location lookup); `welcome_resend` resolves the workspace's `host_name` to the agent's id up front and then sends by id, so the `data.json` name survives only as welcome's input.
